# QuickPaste v0.20 Daily First-Launch Update Check Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Change automatic update throttling from a rolling 24-hour interval to one silent check on the first launch of each local calendar day, strengthen version governance, and publish QuickPaste 0.20.0.

**Architecture:** Keep the existing updater bridge, timestamp key, 15-second startup delay, GitHub Release endpoint, and manual update flow. Add a compatible local-date cache key so time-zone changes cannot reinterpret an earlier attempt, change the pure domain eligibility rule, prove the lifecycle behavior in `App.native-settings.test.ts`, and synchronize governance and release metadata.

**Tech Stack:** Vue 3, TypeScript, Vitest, Tauri 2, Rust, PowerShell, GitHub CLI.

## Global Constraints

- Windows remains the only delivery platform; release NSIS and portable ZIP only.
- Automatic checks are silent on failure and never download or install without user action.
- The attempt timestamp and local-date key are local-only and contain no clipboard content, settings payload, or device identifier.
- Version is exactly `0.20.0` in npm, Cargo, Tauri, release documentation, and lockfiles.
- Use test-first RED/GREEN cycles before production changes.

### Task 2: First launch checks; same-day relaunch skips

**Files:**
- Modify: `src/App.native-settings.test.ts`
- Verify: `src/composables/useUpdater.ts`

**Interfaces:**
- Consumes: existing `quickpaste-update-check-v1` timestamp, the compatible local-date key, and 15-second timer.
- Produces: one automatic `checkForUpdate()` call on the first local-day launch and zero additional automatic calls on a same-day relaunch.

- [ ] **Step 1: Write the failing lifecycle test**

Use fake timers and set the clock to 00:01. Seed the cached timestamp at 23:59 on the preceding local date, mount the app, advance 15 seconds, and expect one check. Unmount, mount again without changing the date, advance 15 seconds, and expect the call count to remain one.

- [ ] **Step 2: Run the lifecycle test and verify RED**

Run: `npm test -- src/App.native-settings.test.ts -t "checks updates on the first launch of each local day"`

Expected: zero calls after the first timer because the old rolling interval considers the previous timestamp too recent.

- [ ] **Step 3: Persist the attempt date without breaking old caches**

Keep the existing timestamp and write an additional `YYYY-MM-DD` local-date key in `finally`; old caches without the key check once to complete migration. Write both keys independently and remove stale date data after a date-write failure. Failures remain silent, and the timer is cleared on unmount.

- [ ] **Step 4: Run the lifecycle test and update suite**

Run:

