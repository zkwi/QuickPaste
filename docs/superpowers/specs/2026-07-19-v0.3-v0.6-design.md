# QuickPaste v0.3–v0.6 设计规格

## 目标

在不引入独立服务、云端同步或随包模型的前提下，把 QuickPaste 从 500 条规模的文本/图片历史升级到可保存文件、多格式富文本并能管理 10,000 条以上历史的 Windows 本地剪贴板工具，最终版本统一定为 `0.6.0`。

## 已批准范围

- v0.3：CF_HDROP 文件/多文件、HTML/RTF 多格式记录、保留格式/纯文本粘贴、类型化操作、增量 SQLite、版本迁移、数量/图片字节/期限容量策略。
- v0.4：SQLite 备份与原子恢复、FTS5 和来源/类型/集合组合筛选、10,000 条基准、损坏恢复、真实磁盘占用。
- v0.5：单层集合、可编辑永久片段、管理页批量操作、固定/集合/搜索统一过滤。
- v0.6：默认开启但可关闭的 Windows 本地 OCR、代码预览按需加载语法高亮。

不复制文件原文，不上传剪贴板正文，不捆绑 OCR/翻译模型或 FFmpeg，不重新引入 MSI，不改变普通权限主进程和一次性管理员粘贴 helper 的安全边界。

## 方案比较

### 方案 A：继续扩展前端全量数组和 JSON payload

优点是改动少。缺点是每次变化仍会深度比较和整表重写，图片会在启动时全部 base64 化，FTS5 只能成为未被产品使用的旁路。该方案无法可信支撑 10,000 条历史，拒绝。

### 方案 B：类型化 SQLite + 增量 mutation + 分页查询（采用）

SQLite 是桌面运行时唯一真值。前端通过窄 IPC 提交 upsert/delete/batch mutation，通过 FTS5 分页查询元数据，只在预览或粘贴时加载大格式正文。该方案保持模块化单体，又能控制写放大、内存和首帧延迟。

### 方案 C：事件日志、独立索引服务或外部模型进程

可扩展性最高，但会增加恢复、部署、权限和维护成本，不符合个人项目当前规模，拒绝。

## 架构边界

### Rust 原生层

- `src-tauri/src/history.rs`：类型化 schema、`application_id`/`user_version` 逐版迁移、增量 mutation、FTS5、容量裁剪、集合、备份/恢复、损坏隔离、存储统计和 10k 基准共用 API。
- `src-tauri/src/clipboard_formats.rs`：Windows 格式包读取/写回。一次打开剪贴板读取 plain/HTML/RTF/CF_HDROP；图片沿用已验证链路。写回先清空一次，再追加同条记录的所有格式。
- `src-tauri/src/ocr.rs`：WinRT `Windows.Media.Ocr` 适配；只处理已进入历史的图片，缺少系统语言包时返回 unavailable，不下载模型。
- `src-tauri/src/system_actions.rs`：显式打开 HTTP(S) 链接、打开/定位文件和保存图片；所有路径/URL 在原生层重新校验。
- `src-tauri/src/metrics.rs`：只记录计数和耗时，不记录正文；提供暖唤起、回贴和捕获完整性的本地验收数据。
- `src-tauri/src/lib.rs`：只做窗口、托盘、会话、helper 和上述模块的编排。现有 helper 协议保持不变。

### TypeScript domain/platform

- `src/domain/clipboard.ts`：扩展为带 `formats`、文件元数据、集合、永久片段和 OCR 状态的运行时可校验模型；富文本仍以 plain text 作为可搜索/可访问回退。
- `src/domain/clipActions.ts`：根据记录类型、可用格式和 surface 生成动作，避免模板、快捷键、右键菜单漂移。
- `src/domain/historyQuery.ts`：统一 query/kind/source/collection/pinned 组合筛选和分页游标。
- `src/domain/collections.ts`：单层集合、永久片段和批量选择纯规则。
- `src/platform/history.ts`：增量 mutation 队列、查询、统计、备份恢复和错误降级。
- `src/platform/clipboard.ts`：`preserve`、`plain`、`files`、`image` 四种明确写回方式。
- `src/platform/ocr.ts`、`src/platform/system.ts`：窄 IPC 适配，不在浏览器/组件中散布 Tauri 调用。

### Vue UI

- 快速面板只新增完成搜索、选择、预览和粘贴所需的类型化行为；集合编辑、批量操作、备份恢复、容量配置和 OCR 管理只进入管理页/设置页。
- `ClipContextMenu.vue` 消费动作规则；文件显示缺失状态，富文本明确显示“保留格式粘贴/纯文本粘贴”。
- `CodePreview.vue` 只有进入代码预览时才动态加载 `highlight.js` core 和命中的语言模块；失败时回退到转义纯文本。
- HTML/RTF 不直接通过不受控 `v-html` 渲染，避免脚本、外部资源请求和正文侧信道。
- 管理列表采用分页，不把 10,000 行一次性渲染到 DOM。

## 数据模型

### `clips`

核心字段：`id`、`kind`、`title`、`plain_text`、`source_app`、`copied_at`、`updated_at`、`pinned`、`permanent`、`collection_id`、`search_terms`、`ocr_text`、`ocr_status`、`content_hash`、`logical_bytes`、展示颜色和图片尺寸。

### `clip_formats`

一条 clip 最多各有一个 `text/html/rtf/image` 格式。HTML 保存 UTF-8，RTF 和图片保存 BLOB。单格式和单条记录有硬上限；超过上限时保留 plain text 并标记被省略格式，不能静默让整次捕获失败。

### `clip_files`

按 `ordinal` 保存路径、名称、扩展名、是否目录、捕获时大小/修改时间/存在状态。只保存路径与元数据，绝不复制源文件。

### `collections`

仅一层：`id/name/created_at/updated_at/sort_order`。clip 只能属于零个或一个集合，删除集合时记录回到“未分类”。

### FTS5

索引 title、plain text、source app、search terms、OCR 和文件名。中文子串使用 bundled SQLite 的 trigram tokenizer；短查询走参数化 `LIKE` 回退。kind/source/collection/pinned/time 使用普通索引，所有 MATCH 输入由后端构造，不能拼接原始用户语法。

## 数据流

1. Windows 监听器收到剪贴板更新消息并核对序列号。
2. 一次打开剪贴板读取完整格式包；生成不含正文日志的格式摘要和内容哈希。
3. 内部写入签名先拦截自身回流；外部格式包通过 Tauri 事件交给 domain 层生成展示字段和拼音索引。
4. 前端立即显示记录，同时把单条 upsert 放入串行 mutation 队列；失败时保留 UI 内容并显示可重试状态，不以空数组覆盖数据库。
5. 图片在 OCR 开启时进入有界本地队列；结果以单字段增量更新并刷新 FTS。
6. 搜索/组合筛选请求带 generation；旧请求不能覆盖新输入。列表只返回一页，大 BLOB 在预览/粘贴时按需读取。
7. 粘贴前按类型组装格式包，写完后重新核对内容摘要和剪贴板序列，再复用现有普通/helper 回贴链路。

## 容量策略

- 默认普通历史上限 10,000 条；设置允许降低，固定项和永久片段不被静默删除。
- 默认图片总字节上限 256 MiB；按最旧、未固定、非永久图片优先裁剪。
- 保留期限继续支持 7/30/90 天/永久，并与数量、图片字节限制同时生效。
- 单 HTML/RTF 8 MiB、单图片 64 MiB、单文件列表 256 项；固定项导致超预算时在存储页明确显示，不自动破坏用户固定内容。
- 存储页同时显示逻辑正文、图片 BLOB、主 DB、WAL、SHM 和总物理字节；删除后不每次 VACUUM，提供显式整理。

## 备份、恢复与损坏

- 备份是可独立打开的 SQLite 文件，明确提示包含未加密的剪贴板正文。
- 导出先写同目录临时文件，校验 `quick_check` 后原子改名。
- 导入先以只读方式检查 application id、schema、行数和 BLOB 上限；再在一个 live DB 事务内替换业务表。任何错误整笔回滚。
- 启动加载遇到 SQLite 损坏时，把原文件及 WAL/SHM 移到带时间戳的 recovery 目录，创建空库并在 UI 暴露恢复路径；不得静默覆盖损坏文件。

## OCR 与高亮

- OCR 设置默认 `true`，显式关闭必须持久化；关闭后不发起新任务。
- OCR 只使用用户 Windows 已安装的语言；不可用、超尺寸和失败分别记录状态。
- 图片哈希去重任务，结果只增加可搜索文字，不修改图片正文。
- 高亮器不进入快速列表 bundle；语言识别失败或动态 import 失败时显示原始 `<pre>`。

## 性能、可靠性和隐私验收

- 暖唤起：Release 包预热 50 次、采样 500 次，快捷键处理到前端首帧 p95 ≤120ms。
- 搜索：固定种子 10,000 条，预热 100 次、1,000 个混合查询，SQLite 和输入到结果首帧 p95 均 ≤50ms。
- 普通回贴：真实普通权限应用至少 10,000 次，总成功率 ≥99.5%。
- 捕获：至少 100,000 次构造外部写入，漏记、额外重复和自身回流之和 <0.1%。
- DPI：100/125/150/175/200/225/250%，混合缩放、负坐标和多显示器均无裁切/错位。
- 网络：允许更新器访问固定 GitHub 端点，但剪贴板正文、编码和摘要不得进入任何网络请求。
- 安装包：解包扫描不得包含 OCR/翻译模型、FFmpeg/ffprobe 或权重文件。

## 测试策略

- 所有行为先写失败测试，再做最小实现。
- domain/platform 用 Vitest 覆盖 schema、动作矩阵、增量 diff、异步 generation、集合和设置迁移。
- Rust 覆盖逐版迁移、事务回滚、FTS 转义/筛选、容量、备份恢复、多格式摘要、OCR 状态和 DPI 几何。
- `src-tauri/examples/history_bench.rs` 提供可重复 10k release 基准，不引入 Criterion。
- UI 变化除 happy-dom 外，用仓库内 Playwright 配置验证 640×440、800×580、浅/深主题和键盘/IME；真实 DPI/多屏/helper 仍记录为 Windows 人工验收。

## 非目标

- 不复制文件内容、不做文件同步或失效文件修复。
- 不做截图工具、云 OCR、翻译、代码运行、代码工具箱、嵌套集合或标签系统。
- 不为其他操作系统提前实现功能。
- 不提交、推送、打标签或发布；GitHub CI 只作推送后的异步复核。
