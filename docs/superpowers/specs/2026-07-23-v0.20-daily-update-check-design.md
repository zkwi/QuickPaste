# QuickPaste v0.20 每日首启检查更新设计

## 目标与范围

QuickPaste 已有手动检查、后台自动检查、GitHub Release 选择、下载安装和完整性校验。本次不新增更新服务或系统任务，只把自动检查的节流口径从“距离上次尝试满 24 小时”改成“每个本地日历日第一次启动时尝试一次”。自动检查继续默认开启，并在应用启动后延迟 15 秒执行，避免与历史数据库、剪贴板监听和窗口初始化争抢启动资源。用户当天手动检查过更新，也视为当天已经尝试；同日后续启动不再自动请求，但手动按钮始终可用。

最近一次尝试继续保留现有 `quickpaste-update-check-v1` 时间戳，并新增检查发生时生成的本地日期键；旧缓存没有日期键时安全地额外检查一次并完成迁移，不迁移设置 schema。纯领域规则比较记录的本地日期与当前本地日期：同一日期跳过，不同日期立即允许，即使相隔不足 24 小时；日期键写入后的尝试不会因随后切换时区而被重新解释。两个缓存键独立写入，日期键失败时尽力清除旧值并保持 fail-open。自动请求在开始后无论成功、无更新或网络失败，都会记录本次尝试，保证可靠存储下同一天不反复打扰 GitHub。网络失败保持静默，发现新版本沿用现有非抢焦点通知。

## 版本与治理

本功能改变启动行为并提升更新可达性，版本从 0.19.3 提升到 0.20.0。`AGENTS.md`、`CONTRIBUTING.md` 与 `docs/release.md` 同步强化版本治理：新增用户可感知能力默认提升 minor；成组体验改进不得压成 patch；在 `1.0.0` 前允许维护者依据影响主动提高 minor，达到稳定交付标准时应考虑提升到 `1.0.0`，patch 只保留给边界明确的修复。

## 验证

领域测试覆盖同日本地日期、跨午夜不足 24 小时、未来日期、无效缓存，以及切换时区后不能重新解释上次检查日期。应用测试使用假时钟启动两次：跨日第一次启动在 15 秒后检查并写入时间戳与本地日期键，同日第二次启动不重复检查；手动检查后同日重启也不自动重复请求。发布前运行 `npm ci`、`npm run check`、npm/Rust 依赖审计和 `npm run build:windows`，只发布 Windows x64 NSIS 与绿色版 ZIP；真实网络失败、跨时区和长期启动矩阵继续标记为 `pending real-machine`。
